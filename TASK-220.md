### 220. [x] Make Ctrl+V paste (text + images) work in terminals on Windows

**Status:** Done
**Depends on:** none
**Created:** 2026-06-28

**Description**

On **Windows**, pressing **Ctrl+V** in an agent (or shell) terminal does **not** paste —
neither text nor images reach the running `claude` (or shell) process. On macOS the
equivalent **⌘V** works. Make Ctrl+V paste **text** (and best-effort **images**) into
terminals on Windows, matching the macOS paste experience.

**Root cause (grounded):**

- The terminal is a single pooled xterm.js instance created in
  `src/components/Terminal/terminalPool.ts` (`createHost`, ~lines 117-210). It loads
  `FitAddon`, `WebLinksAddon`, and (main-window) `WebglAddon` — but **no clipboard
  addon and no custom key handler** (`@xterm/addon-clipboard` is not installed; there is
  no `attachCustomKeyEventHandler`).
- The only input path is `term.onData((data) => writeStdin(sessionId, data))`
  (`terminalPool.ts:206-210`). `onData` fires for keyboard-generated bytes and for
  xterm's **own** internal browser-paste handling (the hidden textarea's `paste`
  event).
- **macOS ⌘V works** because WebKit raises a native `paste` event on the focused
  textarea, which xterm handles → text flows through `onData` → `write_stdin`.
- **Windows Ctrl+V fails** because, by the standard Linux/Windows terminal convention,
  xterm treats **Ctrl+V** as the literal control character **`^V` (0x16)** and forwards
  *that* to the PTY — it is **not** a paste. (The Linux/Windows terminal paste chord is
  conventionally Ctrl+Shift+V; plain Ctrl+V is a control code.) So the clipboard is
  never read and nothing pastes.
- There is **no Tauri clipboard plugin** installed (`src-tauri/Cargo.toml` has no
  `tauri-plugin-clipboard-manager`; `package.json` has no
  `@tauri-apps/plugin-clipboard-manager`; `capabilities/default.json` has no
  `clipboard-manager:*` permission). The app's existing **copy** path uses
  `navigator.clipboard.writeText` directly (`store.ts:3565` `copyToClipboard`); there is
  no clipboard **read** anywhere.
- The cross-platform convention in this codebase is the store-cached **`platform`**
  signal (`src/store.ts`, set from `ipc.platform()`), `src/platform.ts` helpers
  (`isWindows`, `kbdHint`), and `event.metaKey || event.ctrlKey` in every shortcut
  handler (e.g. the WebLinks ⌘/Ctrl-click at `terminalPool.ts:168-171`).

**Goal:** intercept the platform-appropriate paste chord in the terminal, read the OS
clipboard, and inject its contents into the PTY — so Ctrl+V pastes on Windows (and the
existing ⌘V keeps working on macOS).

**Decided approach (autonomous — see Notes/ASSUMPTIONS.md):**

1. **Custom key handler in xterm** — add `term.attachCustomKeyEventHandler(...)` in
   `createHost`. On `keydown`, detect a paste chord and, when matched, perform the paste
   ourselves and **return `false`** so xterm does **not** also emit `^V` (0x16) to the
   PTY:
   - **Windows/Linux:** `event.ctrlKey && !event.altKey && !event.metaKey &&
     (event.key === 'v' || event.key === 'V')` — i.e. **Ctrl+V** *and* **Ctrl+Shift+V**
     both paste.
   - **macOS:** leave ⌘V to xterm's native paste (do **not** intercept), and do **not**
     hijack Ctrl+V on macOS (it stays `^V`). Gate via the store `platform` signal
     (`isWindows(platform)`), read inside the handler (terminalPool can read live state
     via `useStore.getState().platform`, like other non-React store consumers).
   - **Ctrl+C must remain unchanged** (it is the SIGINT interrupt the agent relies on) —
     do not remap copy in this task. Copy-on-selection is **out of scope**.
2. **Read the clipboard via the Tauri clipboard-manager plugin** (robust + cross-platform;
   `navigator.clipboard.readText()` is unreliable/blocked under WebView2 on Windows):
   - Add `@tauri-apps/plugin-clipboard-manager` (JS) + `tauri-plugin-clipboard-manager`
     (Rust, in `src-tauri/Cargo.toml`), init the plugin in `src-tauri/src/lib.rs`, and
     grant the needed permissions in `src-tauri/capabilities/default.json`
     (`clipboard-manager:allow-read-text`, and `clipboard-manager:allow-read-image` for
     images). Wrap reads in the typed IPC layer (`src/ipc.ts`) per the
     frontend/backend-boundary convention.
3. **Text paste:** if the clipboard has text, call **`term.paste(text)`** (not raw
   `writeStdin`) so xterm wraps it for **bracketed-paste mode** when the agent has
   enabled it (claude does), preserving multi-line paste correctly. `onData` then
   forwards it to `write_stdin` exactly as a macOS paste does.
4. **Image paste (best-effort, flagged for real-box verification):** if the clipboard
   has an **image** (and no text), read it via the plugin, **write it to a temp PNG**
   (a new bounded Rust command, e.g. `save_clipboard_image() -> Result<String, _>`
   returning the absolute temp-file path), and **paste that file path** into the terminal
   (`term.paste(path)`). Claude Code accepts an **image file path** as input and attaches
   the image — this is independent of however Claude reads the clipboard internally, and
   because we fully intercept the keystroke there is no double-handling. Use a temp dir
   under the OS temp location (cross-platform), a unique filename, and clean up
   opportunistically. **This path cannot be unit-tested on CI (real clipboard + GUI +
   the live `claude` CLI on Windows); verify on a real Windows box and record the result
   in `TRAJECTORY_TO_WINDOWS.md` per the CLAUDE.md untestable-path rule.**

**Out of scope:**

- **Copy** (Ctrl+C / Ctrl+Shift+C / copy-on-selection) — Ctrl+C must stay SIGINT; copy
  affordances are a separate concern. This task is paste-only (the card title is
  "Paste (Ctrl+V)").
- Changing macOS behavior: ⌘V keeps working; Ctrl+V stays `^V` on macOS.
- A right-click terminal context menu.
- Any new in-terminal UI; this is keyboard-only.

**Cross-platform (hard requirement):** the paste handler is gated to the Windows arm via
the `platform` signal while macOS keeps its native ⌘V — both OSes paste. The clipboard
plugin and the temp-file command are cross-platform. No hardcoded paths (use the OS temp
dir), no POSIX-only shell-outs. The handler uses the established `platform`/`isWindows`
convention. Walk the change against **both** OSes: confirm macOS ⌘V text paste still
works and Ctrl+V still sends `^V` on macOS; confirm Windows Ctrl+V now pastes text and
does **not** leak a stray `^V`.

**Notes**

- **Autonomous decisions (user not answering; logged in `ASSUMPTIONS.md`):**
  - *Read mechanism:* the **Tauri clipboard-manager plugin**, not
    `navigator.clipboard.readText()` (the latter is unreliable/permission-gated in
    WebView2). Adds a dependency + capability — the smallest robust option.
  - *Paste injection:* **`term.paste()`** (respects bracketed paste) over raw
    `writeStdin`.
  - *Chord:* intercept Ctrl+V **and** Ctrl+Shift+V on Windows; leave macOS ⌘V native and
    Ctrl+V as `^V`. **Ctrl+C stays SIGINT** (copy out of scope).
  - *Images:* save to a temp PNG and paste the **file path** (Claude accepts image
    paths) — chosen because it does not depend on Claude's internal clipboard mechanism
    and, since we fully intercept the key, can't double-handle. Flagged for real-box
    verification.
- **Assumption:** the live `claude` CLI on Windows attaches an image when given its file
  path in the prompt (consistent with its documented image-paste / drag-drop support).
  If real-box testing shows Claude instead needs a different signal (e.g. it reads the OS
  clipboard on a specific keystroke), the implementer should adjust the image path to
  forward whatever Claude actually consumes — the **text** paste fix stands regardless.
- **Depends on: none** — this is a self-contained terminal fix; it *adds* its clipboard
  dependency. No open task produces a prerequisite.
